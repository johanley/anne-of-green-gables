The Anne of Green Gables + Git Project

  • Goal: proof-read and compare editions of Anne of Green Gables, by L. M. Montgomery
  • Project Status: just starting

Why?

  • it's a good novel, worthy of detailed examination
  • at least some of the widely available digitized texts don't seem to be of high quality
  • it's interesting to see how the various editions differ from each other
  • it's interesting to use the power of git to precisely track the variations of a literary text
  • a text of acceptable quality can be used to print your own copy using lulu.com
  • it may help improve the quality of digitized versions of the novel

Interesting remarks from the critical edition of Rubio and Waterston:

  • regarding Page's 40th impression, published in 1916: "no other copy-text can claim to represent her final intentions as much as this 1908 edition."
  • "Montgomery gave no personal input into either of these completely new 1925 editions."
  • regarding the two new 1925 editions: "However, a careful comparison of the newly set Harrap and L. C. Page texts (which are very different in themselves) reveals a common line of descent—not from the manuscript but from the lost typescript."

Sources of Digitized Text

  • Faded Page - current version as of 2008-05-09. (Companion site to Distributed Proofreaders Canada.)
  • Wikisource
  • Project Gutenberg - current version as of 2016-10-06. They usually get texts from Distributed Proofreaders, but not always.
Issues:
  • sometimes they aren't explicit about the copy-text used as source.
  • basic facts are often unclear, or hard to find: How many versions have there been? What were the changes between versions? How many people have edited the text?
  • in short, there's no way to evaluate the quality of the text.

UTF-8 Encoding Everywhere

All text files in this project use the UTF-8 encoding. If you see odd characters on your screen, check that your tools are using the UTF-8 encoding.

Standardized Line Breaks

The git tool compares files line by line, and character by character. It has no knowledge of what a sentence is.

It can easily compare text that has the same line breaks, say

I don't deny there's something in what you say, Rachel. I've had some qualms myself.

versus the slightly different text:

I do not deny there's something in what you say, Rachel. I've had some qualms myself.

But it can't effectively compare text that has different line breaks:

I don't deny there's something in what you say, Rachel. I've had some qualms myself.

versus:

I do not deny there's something in what you say, Rachel. I've had some qualms myself.

With such text, git will find a huge number of trivial differences just because of the line breaks.

With two independent sources of a text, it's extremely likely that the line-breaks will different. The work-around is to enforce a policy: each line is a complete sentence.

Text Editor Recommendations

In this project, it's recommended that your text editor software:
  • should have menu actions for saving changes to the git repository (it should know about git)
  • must use UTF-8 encoding (check the settings)
  • must not have an auto-correction tool
  • can apply line-wrap to text, according to the width of the current window, without saving new line breaks to the underlying file
  • should be able to indicate invisible characters (spaces and line breaks)
Here's a screen shot of the Eclipse code editor being used to proofread text versus a scanned version of the book on archive.org. Note how it is wrapping the long sentences in the window, and showing special symbols for spaces and line breaks.

Two github Repositories

This project has two github repositories, not one:
  • anne-of-green-gables: a repo for documentation (what you're reading now), with one branch only.
  • anne-of-green-gables-editions: a repo for the various editions of Anne of Green Gables, with one branch for each edition.
Branching can only be done at the top level, the level of the repository. It's best to use a structure that reflects the underlying data.

Uncorrected Versus Corrected

Almost every text has items that could be corrected in some way. In this project, corrections to the text are handled by branches:
  • the core branch for an edition will have zero corrections applied to it.
  • secondary branches, branched off of the core, are used to apply corrections to the core text.
Some corrections are for egregious and obvious typos, but others are more subtle. The policy of what to apply as a correction, and what not to apply, is a decision that can be made in many different ways. That policy is thus best relegated to a secondary branch, and not to the core data. The core data remains the core data, without ambiguity, while different policies for applying corrections can be applied at will in a secondary branch off of the core.

No Front Matter

This project captures the contents of the 38 chapters of the book. Items outside the chapters, including the table of contents, aren't captured here.

Chapter Title Conventions

The chapter titles are to be placed on the first line of each chapter. The chapter title appears in the file names, but that is for convenience only. The official chapter title is the first line of the text file. For example, if you wish to put a comma in the title of Diana is Invited to Tea With Tragic Results, then just alter the first line of the file for that chapter; do not change the name of the file.
