The Anne of Green Gables + Git Project

Why?

  • to significantly improve the quality of (open-source) transcriptions of the novel
  • to detect transcription errors by doing diffs between different transcriptions
  • to document errors and issues in the first editions
  • to document the variations between editions
  • to print a copy of the novel, using a robust text

You can also use the text to create your own e-book, in epub or mobi format.

Doing diffs with other transcriptions lets a proof-reader stand on the shoulders of others. That is the main point of this project.

Main Results So Far

Result #1: a robust transcription of the uncorrected text of an early edition: 1908 L. C. Page (4th impression). This edition was chosen simply because it is the earliest available on archive.org. It was produced by proof-reading versus this copy-text on archive.org, by one person (John O'Hanley) over a 24 day period, and then (this is the important point) it was diffed with three other independent transcriptions to eliminate many transcription errors.

Result #2: a list of errors/issues with the 1908-LCP-4th text, gathered from critical editions and my own proof-reading.

Result #3: the above list of errors has been applied to the uncorrected text, and is available here as a separate transcription. For most people, this is the most useful item in this project. A raw listing of the actual differences between the uncorrected and corrected versions can be viewed here.

Result #4: I have detected issues in other transcriptions:

  • Distributed Proofreaders of Canada - 33 issues, 30 of which have been corrected as of 2021-01-09. The corrections were applied by Ross Cooling of fadedpage.com.
  • Wikisource - 113 issues, 111 of which have been corrected as of 2021-01-07. The corrections were applied by this project, James Tauber (working independently on his own project), and several Wikisource editors who graciously tracked down a few minor bugs over the course of several days.

Result #5: My transcription was accepted by Project Gutenberg, and published to their site on 2021-01-21. My transcription is now their second transcription (in English) of Anne of Green Gables. The first one has about 1,000 differences with the first edition of the novel. Many of those differences are caused by modernization and Americanization. The remaining differences are still numerous, and show significant, non-trivial changes to the text. My transcription, on the other hand, is much closer to the text of the first edition.

Why use git?

  • it can reflect different transcriptions/editions naturally using different branches
  • it lets you clearly see the variants (the diff) between editions
  • it lets you post improvements after each proof-reading cycle
  • it's a widely used, robust tool, well-suited to the job

Two github Repositories

This project has two github repositories, not one:
  • anne-of-green-gables: a repo for documentation (which you're reading now), with one branch only.
  • anne-of-green-gables-editions: a repo for various transcriptions of Anne of Green Gables, with one branch for each transcription. Branches can also be created for applying corrections/edits to a transcription. Merging between branches is not a typical operation in this project. The main requirement is simply to do diffs between branches.

Lingo

  • copy-text - the exact book which is being transcribed (identified by publisher, year, impression, and so on).
  • transcription - the content of the copy-text, in a text file. The text file is usually proof-read by one or more humans.
  • transcription error - a mismatch between the copy-text and its transcription.
  • branch - in a version control system (such as git), a branch is a separate, parallel stream, containing a sequence of revisions of files. In this project, different copy-texts get different branches for their transcriptions. Each branch can have multiple revisions of its transcription.
  • diff - the result of comparing two revisions of a file (often across different branches). Shows the textual variations, the places where the two revisions differ in any way. Here's an example of a tool showing a diff.

Simple Scheme for Error Detection

From what I can see, the groups on the web that provide transcriptions don't talk to each other. They don't compare their transcriptions to the output of other similar groups.

The idea of this project is simply that doing diffs on independent transcriptions is an easy way to find a large number of errors, and significantly increase accuracy.

Some ad hoc lingo, for the nonce:

  • TE(A) - the transcription errors in a single transcription A.
  • UTE(A, B, C) - the universal transcription errors common to a set of transcriptions A, B, C. That is, the transcription errors repeated in exactly the same way across all members of a set of transcriptions A, B, and C.

Work Flow For Improving My Transcription

First I do a full proof-read of my transcription.

Then, for each of the other transcriptions A, B, C:

  • I do a diff between me and the other transcription
  • for each difference, I find out if it's a real textual difference in the copy-text, or a transcription error. I do this by visually comparing directly with the copy-text itself (for example on archive.org).
  • if it's a transcription error, then I fix one of the transcriptions (in rare cases, both). In this case, I'm removing one item from TE(me) or TE(A). Note that only the non-universal transcription errors are found using this technique.
At the end of the process, what's left over is UTE(A, B, C, me), the universal transcription errors, the ones that are exactly the same in all of the transcriptions.

The UTE(A, B, C, me) errors are more difficult to deal with. They can't be found by doing diffs, since they are present in all of the input transcriptions. Can they be detected using other means? In a few isolated cases, yes. If a policy of Americanization or modernization has been applied to all transcriptions, then that policy can be reversed, in whole or in part, by writing a simple find-replace script to change the transcription. But those are special cases (marked as 'known', in the above diagram).

Rule 1: All non-universal errors can be eliminated by doing diffs.

TE(me) ⇒ UTE(A, B, C, me)

At the end of the process described above, the transcription errors in my output equal the universal transcription errors across all of the inputs. This expresses the benefit of doing diffs.

Rule 2: Independent transcriptions having no common ancestor are best.

It's best that each transcription be as independent from the others as possible:
  • different copy-text
  • different OCR (optical character recognition) machine/software
  • different proof-readers
  • different organizations
Such independence pushes the system in the direction of avoiding universal transcription errors.

Somewhat surprisingly, it's beneficial to even use different editions of the source text. For example, this project uses a transcription of a 1946 edition of Anne of Green Gables to produce a transcription of the 1908 edition (4th impression). This is beneficial because it ensures more independence of transcriptions. It's not a problem, because the real textual diffs between 1908 and 1946 are small, and are detected and accounted for by the work flow described above.

Rule 3: In the end, for a large text, you'll never be sure that you have reduced TE(me) to zero.

A computer can compute an exact result only when all of the data is present. But in this case, one half of the data is in a physical book, outside the computer. That's the problem. Still, you can increase your level of confidence in a transcription by using the above technique.

Distance Between Transcriptions

Two transcriptions can differ from each other for various reasons:
  • transcription errors
  • real differences in the underlying copy-text
  • corrections to real errors in the underlying copy-text
  • minor differences that don't change the meaning of the text (modernizing to-day to today, Americanizing parlour to parlor, hyphenation, and so on)

It would be very unusual if two transcriptions of a long novel were absolutely identical.

Various stats can be used to quantify how different two transcriptions are. For example, here are some stats output by git's diff command. They show the distance between the 1908-LCP-4th branch and this project's initial snapshots of other transcriptions. The smaller the number, the more similar they are.

Policy For Typos/Errors

Almost every copy-text has items that could be corrected in some way (typos and so on). In this project, absolutely all such corrections are handled by using multiple branches in git:
  • the core branch for an edition will have zero corrections for typos applied to it.
  • secondary branches, branched off of the core, are used to apply all corrections to the copy-text, including basic corrections for egregious typos.
Some corrections are for egregious and obvious typos, but other corrections are more subtle. (An example of a subtle correction is to change the spelling of Macpherson to MacPherson.) The policy of what to apply as a correction, and what not to apply, is a decision that can be made in many different ways. That policy is best relegated to a secondary branch, and not to the core data. The core data remains the core data, without ambiguity, exactly matching the copy-text (we hope), while different policies for applying corrections can be applied at will in a secondary branch off of the core.

No Front Matter

The transcription produced by this project captures the contents of the 38 chapters of the book. Items outside the chapters, including the table of contents and dedication, aren't captured here. (That being said, errors in the table of contents of 1908-LCP are noted below.)

Chapter Title Conventions

The chapter titles are to be placed on the first line of each chapter. The chapter title appears in the file names, but that is for convenience only. The official chapter title is the first line of the text file. For example, if you wish to put a comma in the title of Diana is Invited to Tea with Tragic Results, then just alter the first line of the file for that chapter. Don't change the name of the file. That would mess things up.

Known Issues: 1908, 4th impression

There are different types of issues encountered in this text. A small number are serious, some are mejum, but most are weeny-teeny. They are collected here to aid those who wish to edit the text, and produce their own version. For more information, see the modern critical editions listed above.

I have divided these issues into two kinds:

  • errors: the text is either certainly or very likely in error in some way. The idea is to stop the errors from propagating.
  • matters of taste: in my own opinion, changes that I would consider.
Any such division is arbitrary, and opinions will assuredly differ. Nevertheless, I think it's helpful to others to make this kind of separation.

Issues: not actually errors?

Our spirits can comune.
[Ch. 17, p190] The correct spelling is commune, but this probably shouldn't be corrected, since Anne misspells several words in her letter: comune, busum, improoving. Annotated silently changes the spelling to commune, but leaves the spelling of the other two words. In this respect, they are matching the MS. It seems that 1925-GGH changes to 'commune' (example from 1930). It seems that 1925-LCP also changes to 'commune' (example from 1933.) I disagree with these later editions. I think the intent of the author was to intentionally misspell three words in the letter, not two, and that the spelling of the first edition should be preserved.

pail the barge
[Ch. 28, p310] Change to 'pall'. Match the MS. 2007-NCE says that this error in the first impression was actually corrected in the 4th impression, so it's not visible in the above link. But wait! This 1st impression does not have the error. What's going on?

Coincidentally, I found a similar error when reading the novel Germinie Lacerteux. It has this mistake, where the French word elle is poorly printed, and appears as eile instead. Such a misprint would change pall to pail.

Issues: personal taste

In the list below, I list some issues that I personally would like to examine when creating my own version of the text. They are a matter of opinion. Don't take them too seriously. I'm just noting them here for my own benefit. You can take them as an example of the kind of edits someone might make to the core text, according to their taste. These speed bumps are places in the text where, while reading, my attention is distracted from the make-believe world of the story to the real world of thinking about the text's style.

And as for the risk...
[Ch. 01, p10]. Risk...risks...risks: should all three instances of 'risk' be either singular or plural? Colloquial speech.

It is a kind of pretty place.
[Ch. 02, p26]. Change to 'kind of a pretty place'. Match the MS.

That’s Barry’s pond
[Ch. 02, p28]. Capitalize to Pond. This place name occurs elsewhere in a number of places in the text. Might change 'Shore Road' in the same way.

Call you Cordelia!
[Ch. 03, p35]. Change the exclamation mark to a question mark. Why is she yelling? The Annotated Anne of Green Gables makes this change, but they don't note it as a change.

said Marilla sharply, eying her
[Ch. 03, p37]. Change to 'eyeing'. The UK spelling 'eying' just looks odd.

four dark, low-turned posts
[Ch. 03, p39]. Change to 'four dark, low, turned posts'. Match the MS.

Suppose she wasn’t really going to stay here!
[Ch. 04, p43]. Seems completely backward from the intended sense. Why express enthusiasm for the undesired alternative? Move the exclamation point to the next sentence? Italicize 'was'? The author uses an exclamation point here not for expressing a positive feeling, but for expressing a strong feeling. Chapter 31 has a similar sentence: 'Suppose they did not pass!'.

held his tongue,—but this at least was natural,—so that
[Ch. 04, p47]. The commas are superfluous, since the em dashes already do the work.

beating a retreat down cellar after potatoes.
[Ch. 04, p50]. Change to 'down to the cellar'? Strange wording, similar to 'down-stairs', but with two words.

saw that aggravating Matthew
[Ch. 04 , p52]. Ambiguous, poorly worded; what 'that' means is clear only after re-reading the sentence. Saw that 'she was' aggravating?

Mrs. Peter has a large family
[Ch. 06, p63]. The woman is named in 3 ways, in this order: 'Mrs. Peter Blewett', 'Mrs. Peter', and 'Mrs. Blewett'. Could her first name really have been Peter? Or is that her husband's first name? It seems entirely intentional on Montgomery's part. 'Mrs. Peter' reminds me of 'Mrs. Rachel' for Mrs. Rachel Lynde. Chapter 16 has 'Mr. and Mrs. Chester Ross', 'Mrs. Chester Ross'.

picking them— I wouldn’t want
[Ch. 08, p86] Extra space after the em dash; still present in the 39th impression. Is this just an artifact of text justification?

at touch of that thin little hand
[Ch. 10, p107] Change to 'the touch'. Annotated does this.

as plain as waist and skirt and tight as sleeves
[Ch.11 , p109]. Add a comma somewhere? After 'skirt'?

Nothing daunted Anne proceeded onward
[Ch. 11, p112]. Add a comma after 'daunted'? See reference.

General speed bump issue: there are many times when the speech of the characters seems slightly too formal for rural Islanders. For example, contractions such as it is versus it's.

  • if I hadn't had an imagination
  • if I hadn't an imagination (as in the manuscript)
In chapter 15:
  • Marilla, that is the first compliment I have ever had

For Anne in particular, Montgomery is trying to walk a delicate tight-rope: when young, her diction is part rural, part polished. It's hard to know where to draw the line.

Ways To Distinguish Provenance

It's fun to try and guess the provenance of a given text, to see where it might come from. Modern non-critical editions of the book often don't state the source text explicitly. You may want to play detective and guess at the possible origin of the text. This game is uncertain because editions can make use of:
  • Americanization (parlor, not parlour)
  • modernization (today, not to-day)
  • spell-checkers
  • multiple sources for the text
  • arbitrary changes made by an editor

The following applies to early editions only. See 2007-NCE and Annotated for more info.

There is a sentence in chapter 16 which is distinct between three main early editions. It happens when Anne discovers that Diana got drunk on 'raspberry cordial':

  • 1908-LCP: face downward on the sofa in an agony
  • 1925-GGH: face downward on the sofa in agony
  • 1925-LCP : face downwards on the sofa in an agony (same as 1942-RP)
  • 1924-Cornstalk : face downwards on the sofa in an agony (same as 1925-LCP)

1908-LCP and 1925-LCP can also be distinguished using these characteristics:

  • chapter 1: Barnado boy in 1908, and Home boy in 1925
  • chapter 1: Hopetown in 1908, and Hopeton in 1925
  • chapter 17: The presence of comune maps to editions prior to 1925 (I think). Anne's letter has three intentional spelling errors in it: comune, busum, and improoving, as in 1908. Both 1925-GGH and 1925-LCP (I think) revert the spelling of one of the three words to commune. (In modern editions, editors can unwittingly apply spell-checkers to these words.)

1925-LCP has these characteristic errors that identify it:

  • chapter 14: beatification is changed to beautification
  • chapter 14: all ready is changed to already
  • chapter 19: ecstasy is misspelled as ecstacy
I don't know if these 1925-LCP errors were ever corrected. They are still present in 1946-RY. Ryerson used the 1925-LCP plates.

1925-GGH has these characteristics that help identify it:

  • chapter 5: the spelling of the author's name Thompson is corrected to Thomson
  • chapter 2: wondering afar, star-led is changed to wandering afar, star-led (matches MS)
  • chapter 2: yellowish grey wincey is changed to yellowish white wincey
  • chapter 16: beginning to work is changed to beginning to go (matches MS)

The Provenance of Project Gutenberg's 2016 Transcription

The Project Gutenberg 2016 transcription doesn't state its copy-text. Using the above, it seems like it may be a mongrel text, derived from more than one source (modernized, Americanized, and extensively edited), based on the following contradictory observations:
  • chapter 16: face downward on the sofa in an agony (so: LCP)
  • chapter 1: Home boy (so: 1925-LCP)
  • chapter 1: Hopeton (so: 1925-LCP)
  • chapter 17: commune (so: 1925-LCP)
  • chapter 5: Thompson (so: not 1925-GGH)
  • chapter 2: yellowish grey (so: not 1925-GGH)
  • chapter 19: ecstasy (so: not 1925-LCP)
  • chapter 14: beatification (so: not 1925-LCP)

Standardized Line Breaks

The git tool compares files line by line, and character by character. It has no knowledge of what a sentence is.

It can easily compare text that has the same line breaks, say

At first Matthew suggested getting a Barnardo boy. But I said ‘no’ flat to that.

versus the slightly different text:

At first Matthew suggested getting a Barnado boy. But I said ‘no’ flat to that.

But it can't effectively compare text that has different line breaks:

At first Matthew suggested getting a Barnardo boy. But I said ‘no’ flat to that.

versus:

At first Matthew suggested getting a Barnardo boy. But I said ‘no’ flat to that.

With such text, git will find a huge number of trivial differences just because of the line breaks.

With two independent sources of a text, it's extremely likely that the line-breaks will differ. The work-around is to enforce a simple policy: each line is a complete sentence.

Standardized Format

The transcriptions in this project have the following format:
  • one sentence per line (as described above)
  • a new paragraph is represented by a single empty line
  • italics are represented by underscores: _like this_ stands for like this
  • there's no representation of bold text
  • quotations use left-quote and right-quote “”, not vertical quote ": “Hello there,” said...
  • apostrophes use ’, not ': I’ll
  • indentation is never used
  • chapter titles at the start of a file are in all-caps
  • em dash is used (not hyphen)
  • small caps are not represented
  • the files are encoded in UTF-8, so that any glyph can be represented

Text Editor Recommendations

For this project, here's a list of recommendations for your text editor:
  • it should have menu actions for saving changes to the git repository (it should know about git)
  • it must use UTF-8 encoding (check the settings)
  • it must not have an auto-correction tool
  • it can apply line-wrap without hyphenating text on the right margin
  • it can apply line-wrap to text without saving temporary line breaks to the underlying file
  • it should be able to indicate invisible characters (spaces and line breaks)
  • it should be able to show the diff between versions of a file
  • it would be nice if it could show the Unicode designation of a character, for example U+0074 : LATIN SMALL LETTER T; if not, you can use this tool
