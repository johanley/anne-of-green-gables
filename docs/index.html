The Anne of Green Gables + Git Project

Why?

  • to significantly improve the quality of digitized versions of the novel
  • to detect errors by doing diffs between different transcriptions
  • to explore the variations between editions
  • to print a copy of the novel, using a robust text

You can also use the text to create your own e-book, in epub or mobi format.

Why use git?

  • it can reflect different transcriptions/editions naturally using different branches
  • it lets you clearly see the variants (the diff) between editions
  • it lets you post improvements after each proof-reading cycle
  • it's a widely used, robust tool, well-suited to the job

Two github Repositories

This project has two github repositories, not one:
  • anne-of-green-gables: a repo for documentation (which you're reading now), with one branch only.
  • anne-of-green-gables-editions: a repo for various transcriptions of Anne of Green Gables, with one branch for each transcription. Merging between branches is not a typical operation in this project. The main requirement is simply to do diffs between branches.

Lingo

  • copy text - the exact book which is being transcribed (identified by publisher, year, impression, and so on).
  • transcription - the content of the copy text, in a text file. The text file is usually proof-read by one or more humans.
  • transcription error - a mismatch between the copy text and its transcription.
  • branch - in a version control system (such as git), a branch is a separate, parallel stream, containing a sequence of revisions of files. In this project, different copy texts get different branches for their transcriptions. Each branch can have multiple revisions of its transcription.
  • diff - the result of comparing two revisions of a file (often across different branches). Shows the textual variations, the places where the two revisions differ in any way. Here's an example of a tool showing a diff.

Simple Scheme for Error Detection

From what I can see, the groups on the web that provide transcriptions don't talk to each other. They don't compare their transcriptions to the output of other similar groups.

The idea of this project is simply that doing diffs on independent transcriptions is an easy way to find a large number of errors, and significantly increase accuracy.

Some lingo:

  • TE(A) - the transcription errors in a single transcription A.
  • UTE(A, B, C) - the universal transcription errors common to a set of transcriptions A, B, C. That is, the transcription errors repeated in exactly the same way across all members of a set of transcriptions A, B, and C.

Work Flow For Improving My Transcription

First I do a full proof-read of my transcription.

Then, for each of the other transcriptions A, B, C:

  • I do a diff between me and the other transcription
  • for each difference, I find out if it's a real textual difference in the copy text, or a transcription error. I do this by visually comparing directly with the copy text itself (for example on archive.org).
  • if it's a transcription error, then I fix one of the transcriptions (in rare cases, both). In this case, I'm removing one item from TE(me) or TE(A). Note that only the non-universal transcription errors are found using this technique.
At the end of the process, what's left over is UTE(A, B, C, me), the universal transcription errors, the ones that are exactly the same in all of the transcriptions.

The UTE(A, B, C, me) errors are more difficult to deal with. They can't be found by doing diffs, since they are present in all of the input transcriptions. Can they be detected using other means? In a few isolated cases, yes. If a policy of Americanization or modernization has been applied to all transcriptions, then that policy can be reversed, in whole or in part, by writing a simple find-replace script to change the transcription. But those are special cases (marked as 'known', in the above diagram).

Rule 1: All non-universal errors can be eliminated by doing diffs.

TE(me) = UTE(A, B, C, me)

At the end of the process described above, the transcription errors in my output equal the universal transcription errors across all of the inputs. This expresses the benefit of doing diffs.

Rule 2: Independent transcriptions having no common ancestor are best.

It's best that each transcription be as independent from the others as possible:
  • different copy text
  • different OCR (optical character recognition) machine/software
  • different proof-readers
  • different organizations
Such independence pushes the system in the direction of avoiding universal transcription errors.

Somewhat surprisingly, it's beneficial to even use different editions of the source text. For example, this project uses a transcription of a 1946 edition of Anne of Green Gables to produce a transcription of the 1908 edition (4th impression). This is beneficial because it ensures more independence of transcriptions. It's not a problem, because the real textual diffs between 1908 and 1946 are small, and are detected and accounted for by the work flow described above.

Rule 3: In the end, for a large text, you'll never be sure that you have reduced TE(me) to zero.

A computer can compute an exact result only when all of the data is present. But in this case, one half of the data is in a physical book, outside the computer. That's the problem. Still, you can increase your level of confidence in a transcription by using the above technique.

Other Sources of Digitized Versions

  • Faded Page - current version as of 2008-05-09. Companion site to Distributed Proofreaders Canada. Their copy-text is the 1946-11 Ryerson edition (see link above). Seems to be of good quality, but see corrections listed below.
  • Project Gutenberg - current version as of 2016-10-06. They usually get texts from Distributed Proofreaders, but not always. They don't even indicate the copy-text. The quality of this text seems low. The copy-text seems to be some arbitrary edition having substantial differences with 1908.
  • Standard Ebooks - apparently uses Project Gutenberg as source of their text. (I don't know if they have proof-read/edited the text themselves. I suspect not.)
  • Wikisource
Issues:
  • sometimes they aren't explicit about the copy-text used as source.
  • basic facts are sometimes unclear, or hard to find: how many versions of this digitized text have there been? what were the changes between versions? how many people have proof-read the text, over its lifetime?
  • in short, there's no way to evaluate the probable quality of the transcription.

Policy For Typos

Almost every transcription has items that could be corrected in some way (typos and so on). In this project, absolutely all corrections to the core text are handled by using multiple branches in git:
  • the core branch for an edition will have zero corrections for typos applied to it.
  • secondary branches, branched off of the core, are used to apply all corrections to the core text, including basic corrections for egregious typos.
Some corrections are for egregious and obvious typos, but other corrections are more subtle. (An example of a subtle correction is to change the spelling of Macpherson to MacPherson.) The policy of what to apply as a correction, and what not to apply, is a decision that can be made in many different ways. That policy is best relegated to a secondary branch, and not to the core data. The core data remains the core data, without ambiguity, exactly matching the copy-text (we hope), while different policies for applying corrections can be applied at will in a secondary branch off of the core.

No Front Matter

This project captures the contents of the 38 chapters of the book. Items outside the chapters, including the table of contents and dedication, aren't captured here.

Chapter Title Conventions

The chapter titles are to be placed on the first line of each chapter. The chapter title appears in the file names, but that is for convenience only. The official chapter title is the first line of the text file. For example, if you wish to put a comma in the title of Diana is Invited to Tea with Tragic Results, then just alter the first line of the file for that chapter. Don't change the name of the file. That would mess things up.

John's Speed Bumps

In the list below, I list some issues that I personally would like to examine when creating my own version of the text. They are a matter of opinion. Don't take them too seriously. I'm just noting them here for my own benefit. You can take them as an example of the kind of edits someone might make to the core text, according to their taste. These speed bumps are places in the text where, while reading, my attention is distracted from the make-believe world of the story to the real world of thinking about the text's style.

And as for the risk...
[Ch. 01, p10]. Risk...risks...risks: should all three instances of 'risk' be either singular or plural? Colloquial speech.

It is a kind of pretty place.
[Ch. 02, p26]. Change to 'kind of a pretty place'. Match the MS.

That’s Barry’s pond
[Ch. 02, p28]. Capitalize to Pond. This place name occurs elsewhere in a number of places in the text. Might change 'Shore Road' in the same way.

Call you Cordelia!
[Ch. 03, p35]. Change the exclamation mark to a question mark. Why is she yelling? The Annotated Anne of Green Gables makes this change, but they don't note it as a change.

said Marilla sharply, eying her
[Ch. 03, p37]. Change to 'eyeing'. The UK spelling 'eying' just looks odd.

four dark, low-turned posts
[Ch. 03, p39]. Change to 'four dark, low, turned posts'. Match the MS.

Suppose she wasn’t really going to stay here!
[Ch. 04, p43]. Seems completely backward from the intended sense. Why express enthusiasm for the undesired alternative? Move the exclamation point to the next sentence? Italicize 'was'?

held his tongue,—but this at least was natural,—so that
[Ch. 04, p47]. The commas are superfluous, since the em dashes already do the work.

beating a retreat down cellar after potatoes.
[Ch. 04, p50]. Change to 'down to the cellar'? Strange wording, similar to 'down-stairs', but with two words.

saw that aggravating Matthew
[Ch. 04 , p52]. Ambiguous, poorly worded; what 'that' means is clear only after re-reading the sentence. Saw that 'she was' aggravating?

Mrs. Peter has a large family
[Ch. 06, p63]. The woman is named in 3 ways, in this order: 'Mrs. Peter Blewett', 'Mrs. Peter', and 'Mrs. Blewett'. Could her first name really have been Peter? Or is that her husband's first name? It seems entirely intentional on Montgomery's part. 'Mrs. Peter' reminds me of 'Mrs. Rachel' for Mrs. Rachel Lynde.

as plain as waist and skirt and tight as sleeves
[Ch.11 , p109]. Add a comma somewhere? After 'skirt'?

Nothing daunted Anne proceeded onward
[Ch. 11, p112]. Add a comma after 'daunted'? See reference.

nor ‘Midian,’ either, but it sounds
[Ch.11, p115]. Remove the comma after Midian? Not in MS.

Anne rose to her feet, with clasped hands, the tears
[Ch. 12, p118]. Remove the comma?

in from the yard, eyes shining,
[Ch. 13, p126]. Printing defect. The comma is barely visible; make sure it's there.

between our farm and Mr. Barry’s.
[Ch. 13, p129]. Should the statement be a question?

There’s another, not a sister,
[Ch. 19, p215]. The original text has no italics for 'not'.

General speed bump issue:
There are many times when the speech of the characters seems slightly too formal for rural Islanders. For example, contractions such as it is versus it's, and (in chapter 5) if I hadn't had an imagination versus if I hadn't an imagination (as in the manuscript). For Anne in particular, Montgomery is trying to walk a delicate tight-rope: when young, her diction is part rural, part polished. It's hard to know where to draw the line.

Standardized Line Breaks

The git tool compares files line by line, and character by character. It has no knowledge of what a sentence is.

It can easily compare text that has the same line breaks, say

At first Matthew suggested getting a Barnardo boy. But I said ‘no’ flat to that.

versus the slightly different text:

At first Matthew suggested getting a Barnado boy. But I said ‘no’ flat to that.

But it can't effectively compare text that has different line breaks:

At first Matthew suggested getting a Barnardo boy. But I said ‘no’ flat to that.

versus:

At first Matthew suggested getting a Barnardo boy. But I said ‘no’ flat to that.

With such text, git will find a huge number of trivial differences just because of the line breaks.

With two independent sources of a text, it's extremely likely that the line-breaks will differ. The work-around is to enforce a simple policy: each line is a complete sentence.

Text Editor Recommendations

If you're making changes directly to the files, it's recommended that your text editor software:
  • should have menu actions for saving changes to the git repository (it should know about git)
  • must use UTF-8 encoding (check the settings)
  • must not have an auto-correction tool
  • can apply line-wrap without hyphenating text on the right margin
  • can apply line-wrap to text without saving temporary line breaks to the underlying file
  • should be able to indicate invisible characters (spaces and line breaks)
  • should be able to show the diff between versions of a file
